# Spec Index Generator

Scan a spec directory and produce an ownership registry that maps each file to the concepts it canonically owns.

## Input

The spec directory path. If not provided, ask the user. Common locations:
- `specs/`
- `docs/architecture/`
- `docs/specs/`

## Process

### 1. Discover Spec Files

Recursively find all `.md` files in the spec directory. Exclude:
- `README.md` files (these are section indexes, not concept files)
- `spec-index.md` itself (the output file)
- Files in hidden directories (`.cache/`, `.git/`)

### 2. Extract Ownership from Each File

For each spec file, extract the SSoT ownership declaration using this priority:

**Priority 1 — SSoT header blockquote:**
```markdown
> **Single source of truth** for [concept description].
```
Extract the text after "for" as the ownership declaration.

**Priority 2 — H1 title + first paragraph:**
If no SSoT blockquote exists, use the H1 heading as the concept name and the first paragraph (before any `---` or second heading) as the description.

**Priority 3 — Filename inference:**
If the file has no H1 or summary, derive the concept from the filename: `kebab-case-name.md` → "kebab case name". Flag this file as **missing SSoT header** in the report.

### 3. Detect Issues

While scanning, detect and report:

| Issue | Severity | Description |
|-------|----------|-------------|
| **Duplicate ownership** | CRITICAL | Two files claim ownership of the same concept |
| **Missing SSoT header** | HIGH | File has no SSoT blockquote — ownership was inferred |
| **Orphan file** | MEDIUM | File not listed in any section README.md |
| **Broken cross-reference** | MEDIUM | File references a non-existent file or heading |
| **Overlapping scope** | HIGH | Two files' ownership descriptions overlap significantly |

### 4. Generate the Index

Produce a markdown file with this structure:

```markdown
<!-- Auto-generated by /pmp:spec-index. Do not edit manually — regenerate with /pmp:spec-index. -->

# SSoT Index

> Registry of which spec file owns which concern. Agents and skills read this file to locate the canonical source for any concept without scanning all spec files.

| File | Owns |
|------|------|
| [path/to/file.md](path/to/file.md) | [ownership description] |
| [path/to/other.md](path/to/other.md) | [ownership description] |
```

**Sorting:** Group by section directory, then alphabetically within each section.

**Link format:** Relative paths from the spec-index.md location.

### 5. Report Issues

If any issues were detected in step 3, append an issues section:

```markdown
## Issues

| Severity | File | Issue |
|----------|------|-------|
| CRITICAL | [file.md](file.md) | Duplicate ownership with [other.md](other.md) — both claim "concept X" |
| HIGH | [file.md](file.md) | Missing SSoT header — ownership inferred from filename |
```

### 6. Present and Save

Present the generated index and issues to the user. Ask:
1. **Save as-is** — write `spec-index.md` to the spec directory root
2. **Fix issues first** — address the reported issues before saving
3. **Cancel** — don't save

If saving, write the file and confirm the path.

## How Other Sub-Commands Use the Spec Index

Once generated, the spec index is consumed by:

- **pmp:plan** — Before generating a plan from specs, reads the index to locate canonical sources for each concept. Plan features reference the canonical file, not duplicates.
- **pmp:plan-review** — During spec alignment gate, uses the index to verify plan features reference canonical sources.
- **pmp:spec-review** — During discovery phase, uses the index instead of scanning all files. Detects index staleness.
- **pmp:execute** — Implementer agents read the index to find the authoritative spec for their task.
- **pmp:arc42** — Reads the index as input for reorganization. Produces an updated index after reorganization.
